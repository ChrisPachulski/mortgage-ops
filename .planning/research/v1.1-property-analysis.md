# v1.1 Property Analysis Mode — Research

## Summary

v1.1 wires a new top-level skill mode (`property`) that accepts a Zillow URL,
extracts a `PropertyListing` Pydantic record, runs the full v1.0 calc engine
against `(listing, household, profile)`, and emits a one-page underwriting
workup with a GO / WATCH / NO-GO verdict. The math layer is already shipped:
amortize + affordability + arm_simulate + refi_npv + stress + points +
apr_reg_z + the regulatory predicate library are all available as black-box
CLIs. v1.1 is composition + ingestion + a new report formatter; it does
not add new calc primitives.

The hardest unknown is Zillow ingestion. WebFetch (Claude Code's built-in)
fetches raw HTML to ~10MB, converts to markdown via Haiku, and returns a
prompted summary — it does NOT execute JS, but Zillow embeds essentially
all listing data in a `<script id="__NEXT_DATA__">` JSON blob that ships
in the initial HTML, so a Haiku prompt of the form "extract price, tax,
HOA, beds, baths, sqft, zip, year built, zestimate as JSON" against the
raw page is reliable for the happy path. Failure modes are real
(PerimeterX captcha, 403 from datacenter IPs, sporadic A/B-rendered card
variants) so the architecture MUST be hybrid: WebFetch first, then user
confirms the extracted JSON in a single round-trip, then user supplies
any missing critical field. Paid scraper APIs (Apify
maxcopell/zillow-detail-scraper at $1.70/1k results, ScrapingBee, Bright
Data) are documented for v1.2 only — v1.1 ships interactive gap-fill,
NOT an external API dependency.

The analysis pipeline is "fan-out across program × down-payment, then
synthesize a verdict." For one property the pipeline computes ~24-30
amortization runs (4 programs × 6 down-payment scenarios) plus 1
affordability evaluation per program × 1 stress sweep × 1 points
breakeven (if discount points are in profile.yml) × 1 refi opportunity
snapshot using `FRED MORTGAGE30US - 100bps` as the indicative refi rate.
Total wall-clock under one second per property because the matrix runs
in-process (single Python invocation; not 24 subprocesses). DuckDB gets
a new `analyzed_listings` table joined to `loans` + `scenarios` so
historical analyses are queryable ("show every property I've analyzed in
the last 90 days with verdict=GO and price<$900k") — exposed in v1.2 as
a watchlist mode.

## Patterns (decisions ready to lock)

### Pattern 1: Zillow HTML extraction strategy

**Primary path: Claude Code WebFetch + `__NEXT_DATA__` extraction prompt.**

WebFetch is the only ingestion mechanism that ships with Claude Code and
requires zero new dependencies. It fetches raw HTML locally (Axios under
the hood; no JS execution), converts to markdown, runs a Haiku sub-call
against the content with the caller's prompt, and returns the Haiku
response. Max fetch ~10MB, truncates to 100KB of text for the secondary
prompt.

Because Zillow listing pages are Next.js-rendered, the entire property
record is embedded in a `<script id="__NEXT_DATA__" type="application/
json">` tag that ships in the initial HTML — no JS execution required.
The canonical access path inside that JSON is:

```
data.props.pageProps.componentProps.gdpClientCache →
  JSON-string-wrapped HomeDetails record keyed by
  "ForSaleShopperPlatformFullRenderQuery{...}"
```

**Recommended flow (mode body in `modes/property.md`):**

1. Detect a Zillow URL in the user prompt via substring match
   `zillow.com/homedetails/` or `zillow.com/b/` (both supported — the
   `/b/` form is the building/MLS-listing variant and follows the same
   `{slug}/{zpid}_zpid/` structure).
2. Claude invokes WebFetch with the user-provided URL (URLs cannot be
   dynamically constructed per security policy: "URLs must come from
   the user or from a prior WebFetch/WebSearch result") and a strict
   extractor prompt (sample below).
3. Claude parses the WebFetch response as JSON and validates against
   `PropertyListing` Pydantic model via `scripts/property_analyze.py
   --validate-only --input <tempfile>`. Validation errors come back in
   the standard 6-key Pydantic envelope on stderr (D-19 contract from
   Phase 3 / 10), which Claude narrates to the user.
4. For any field that came back null, empty, or "estimated" without
   provenance, Claude prompts the user:
   > "I couldn't extract the HOA fee from the listing. Is it $0 (no
   > HOA) or a specific number? Type the dollar amount or 'none'."
5. Per the math-discipline contract, every user-supplied value is
   tagged with `provenance: "user_provided"` in the PropertyListing
   record so the audit trail remains intact when persisted to DuckDB.

**Sample WebFetch extractor prompt (lock into `modes/property.md`):**

```
Extract structured property data from this Zillow listing. The page
contains a <script id="__NEXT_DATA__"> JSON blob. Find it, parse it,
and return ONLY a JSON object with these fields (use null for missing
values, never invent values):

{
  "zpid": <integer>,
  "address": "<street + city + state + zip>",
  "zip": "<5-digit>",
  "list_price": "<decimal as string, e.g. '789000.00'>",
  "zestimate": "<decimal as string or null>",
  "property_tax_annual": "<decimal as string or null>",
  "hoa_monthly": "<decimal as string or null>",
  "insurance_estimate_annual": "<decimal as string or null>",
  "beds": <number or null>,
  "baths": <number or null>,
  "sqft": <integer or null>,
  "lot_sqft": <integer or null>,
  "year_built": <integer or null>,
  "property_type": "<one of: SingleFamily, Condo, Townhouse,
    Multifamily, Manufactured, Cooperative, Unknown>",
  "days_on_market": <integer or null>,
  "list_date": "<YYYY-MM-DD or null>",
  "county_name": "<string or null>",
  "state_fips": "<2-char string or null>",
  "county_fips": "<3-char string or null>"
}

If you see substrings like "px-captcha", "Press & Hold to confirm",
or "perimeterx", return {"_block_detected": true, "signal_phrase":
"<which matched>"}.

If __NEXT_DATA__ was truncated before you reached it, return
{"_truncated": true}.

Money fields MUST be Decimal-safe JSON strings (Pydantic v2 strict
mode rejects floats). Output ONLY the JSON object, no prose, no fences.
```

**Failure detection:**

- HTTP 403 → captcha / PerimeterX block. WebFetch surfaces fetch-fail;
  mode narrates: "Zillow returned 403 (bot block). Paste listing
  details manually."
- HTML 200 but body is a captcha (`px-captcha`, `Press & Hold`,
  `perimeterx.net`) → extractor returns `_block_detected:true`; mode
  switches to manual.
- Truncated `__NEXT_DATA__` (`_truncated:true`) → same fallback.

**Why not Playwright / requests + BS4 directly:** adds dep, requires
headless-browser binary in CI, still blocked by PerimeterX without
residential proxies, duplicates WebFetch. Stick with WebFetch.

**URL pattern support:** both `zillow.com/homedetails/{slug}/{zpid}_zpid/`
and `zillow.com/b/{zpid}_zpid/` contain the same `__NEXT_DATA__`. ZPID
extraction regex:

```python
ZPID_RE = re.compile(r"/(\d+)_zpid/?$")
```

Confidence: MEDIUM-HIGH on the happy path (multiple sources confirm
`__NEXT_DATA__` is the canonical access pattern circa 2025; Zillow may
rotate it). Gap-fill fallback is HIGH confidence — no external
dependency.

### Pattern 2: PropertyListing Pydantic schema

**Module:** `lib/property_listing.py` (new file, mirrors `lib/models.py`
shape — `ConfigDict(strict=True, frozen=True, extra="forbid")`).

```python
from datetime import date, datetime
from decimal import Decimal
from typing import Literal
from pydantic import BaseModel, ConfigDict, Field, field_validator
from lib.models import Money  # reuse Phase 1 Money alias

PropertyType = Literal[
    "SingleFamily", "Condo", "Townhouse", "Multifamily",
    "Manufactured", "Cooperative", "Unknown",
]

Provenance = Literal[
    "zillow_scraped",     # came from __NEXT_DATA__ extraction
    "user_provided",      # human-typed in gap-fill
    "zillow_estimated",   # Zestimate / tax estimate (carries
                          # estimated:true label downstream)
    "missing",            # explicitly null after gap-fill confirm
]


class ProvenancedMoney(BaseModel):
    model_config = ConfigDict(strict=True, frozen=True, extra="forbid")
    value: Money | None
    provenance: Provenance


class PropertyListing(BaseModel):
    model_config = ConfigDict(strict=True, frozen=True, extra="forbid")

    # MUST-HAVE (analysis fails loud if missing)
    source_url: str = Field(min_length=10)
    zpid: int = Field(ge=1)
    list_price: Money     # required; no provenance wrapper
    zip: str = Field(pattern=r"^\d{5}$")

    # HIGH (analysis usable but degraded if missing — surface warnings)
    property_tax_annual: ProvenancedMoney
    hoa_monthly: ProvenancedMoney
    insurance_estimate_annual: ProvenancedMoney
    property_type: PropertyType
    beds: int = Field(ge=0, le=20)
    baths: Decimal = Field(ge=Decimal("0"), le=Decimal("20"))
    sqft: int | None = Field(ge=100, le=50000, default=None)

    # LOW (surface in narration only; not used in math)
    address: str | None = None
    lot_sqft: int | None = None
    year_built: int | None = Field(ge=1700, le=2100, default=None)
    zestimate: Money | None = None
    days_on_market: int | None = None
    list_date: date | None = None
    county_name: str | None = None
    state_fips: str | None = Field(pattern=r"^\d{2}$", default=None)
    county_fips: str | None = Field(pattern=r"^\d{3}$", default=None)

    # AUDIT
    fetched_at: datetime
    extraction_method: Literal["webfetch", "manual", "apify"]

    @field_validator("baths")
    @classmethod
    def _baths_half_step(cls, v: Decimal) -> Decimal:
        if (v * 2) % 1 != 0:
            raise ValueError(f"baths must be 0.5 step; got {v}")
        return v
```

**Critical decisions:**

- `list_price` is the ONLY required money field. Tax / HOA / insurance
  all use `ProvenancedMoney` because for new construction or condos
  with pending HOAs, these fields legitimately come back null.
  Downstream code branches on `provenance` to decide whether to inject
  defaults (e.g. `1.1% × list_price` for insurance if missing — see
  Pattern 9 YAML).
- `condecimal` form not used directly — already factored into the
  `Money` alias from Phase 1. `frozen=True` matches every other model
  in the codebase; listing records are append-only into DuckDB.
- `property_type` is a strict Literal because downstream eligibility
  predicates (FHA, VA, Conventional) branch on it.
- ZPID is INT — Zillow IDs are documented numeric; largest observed
  (`2064142765`) fits Python int comfortably.

Confidence: HIGH (mirrors Phase 1 / 4 / 8 model patterns exactly; no
new conventions).

### Pattern 3: Multi-program eligibility comparison

**Module:** `lib/property_analysis.py` (new). Composes existing
primitives; adds no new math.

Algorithm sketch:

```python
Program = Literal["conventional_30", "conventional_15", "fha_30",
                  "va_30", "jumbo_30"]

DOWN_PAYMENT_SCENARIOS = [Decimal(d) for d in
    ("0.035",  # FHA minimum
     "0.05", "0.10", "0.15",
     "0.20",   # PMI-free conventional
     "0.25")]


def analyze_property(listing, household, profile,
                     market_rate_30yr, market_rate_15yr):
    # Step 1: classify eligible programs via existing predicate
    classification = lib.rules.loan_type.classify(
        amount=listing.list_price,
        state_fips=listing.state_fips, county_fips=listing.county_fips,
        property_type=listing.property_type,
        unit_count=1 if listing.property_type != "Multifamily" else 2)
    programs = []
    if classification["conforming"]:
        programs.extend(["conventional_30", "conventional_15"])
    if classification["fha_eligible"]: programs.append("fha_30")
    if profile.va_eligible and listing.property_type in {"SingleFamily",
            "Townhouse", "Multifamily"}:
        programs.append("va_30")
    if classification["jumbo"]: programs.append("jumbo_30")

    # Step 2: 24-cell matrix — amortize + affordability per cell
    cells = [_run_one_cell(listing, household, profile, prog, dp,
                           market_rate_30yr, market_rate_15yr)
             for prog in programs for dp in DOWN_PAYMENT_SCENARIOS]

    # Step 3: stress sweep against headline (default 20%-down conv30;
    #         profile.headline_override to pin elsewhere)
    headline = _pick_headline(cells, profile)
    stress = run_stress_sweep(baseline=headline.amortize_request,
        rate_shocks=[Decimal("0.01"), Decimal("0.02")],
        income_shocks=[Decimal("0.25")])

    # Step 4: refi opportunity snapshot at MORTGAGE30US - 100bps
    refi = refi_npv(... headline + (market_rate_30yr - 0.01) ...)

    # Step 5: points breakeven if profile.consider_points
    points_snap = (points_breakeven(headline.amortize_request,
                                     profile.points_offered)
                   if profile.consider_points else None)

    # Step 6: synthesize verdict (Pattern 4)
    verdict = _synthesize_verdict(cells, stress, refi)
    return PropertyAnalysisReport(listing, classification, cells,
        headline, stress, refi, points_snap, verdict,
        generated_at=datetime.utcnow())
```

**Reuse, not reimplement:** every function in the right-hand side is
already shipped. The new code is composition + a 24-row result matrix +
verdict logic. No new calc primitives.

**Per-program rate selection:** 30yr cells use `market_rate_30yr`; the
15yr cell uses `market_rate_15yr`. FHA and VA pricing differs from
conventional in practice but v1.1 uses the same FRED MORTGAGE30US rate
plus a documented `program_rate_adjustment` knob in profile.yml
(default 0; surface as Open Question #Q4). Intentionally naive —
pricing each program off published rate sheets is out of scope.

**Computation budget:** 4 programs × 6 down-payment scenarios = 24
cells. Each cell = one amortize + one affordability call. The 24-cell
matrix runs IN-PROCESS (single `scripts/property_analyze.py`
invocation that imports all primitives directly); not 24 subprocesses.
Wall-clock well under one second.

Confidence: HIGH (composition only; underlying primitives are already
test-pinned to golden values).

### Pattern 4: Verdict synthesis (GO / WATCH / NO-GO)

**Module:** `lib/property_analysis._synthesize_verdict()`.

```python
Verdict = Literal["GO", "WATCH", "NO_GO"]


def _synthesize_verdict(cells, stress, refi):
    reasons = []
    severity: Verdict = "GO"

    # NO_GO triggers (any one trips it)
    if all(c.blocked_by is not None for c in cells):
        return VerdictResult(level="NO_GO",
            headline_reason="No program qualifies at any down-payment "
                            "scenario",
            details=[c.blocked_by for c in cells if c.blocked_by])
    if min(c.dti_back for c in cells) > Decimal("0.50"):
        return VerdictResult(level="NO_GO",
            headline_reason="Back-end DTI > 50% at every scenario",
            details=[...])

    # WATCH triggers (downgrade GO → WATCH)
    if any(c.dti_back > Decimal("0.43") for c in cells if
            c.program == "conventional_30"
            and c.down_pct >= Decimal("0.20")):
        severity = "WATCH"
        reasons.append("Back-end DTI > 43% (ATR/QM heuristic)")
    if any(s.payment_increase_pct > Decimal("0.30")
            for s in stress.rate_shock_results):
        severity = "WATCH"
        reasons.append("100bps rate shock raises payment >30%")
    if any(s.dti_back > Decimal("0.45")
            for s in stress.income_shock_results):
        severity = "WATCH"
        reasons.append("25% income shock breaches DTI 45%")
    if cells[0].cash_to_close > household.liquid_reserves * Decimal("0.80"):
        severity = "WATCH"
        reasons.append("Cash-to-close consumes >80% of liquid reserves")

    if severity == "GO":
        reasons.append("All programs qualify; stress envelope intact; "
                       "cash reserves comfortable.")
    return VerdictResult(level=severity, headline_reason=reasons[0],
                         details=reasons[1:])
```

**Design notes:**

- NEVER emit a verdict without a reason list. The mode narration leads
  with the verdict and the headline reason; the details flow into a
  bullet block (Pattern 7 report template).
- Thresholds are personal-use heuristics; live in
  `data/reference/property-analysis-heuristics.yml` (NEW YAML; same
  `source:` + `effective:` convention as Phase 2). Lets the user tune
  boundaries without code changes, matching Phase 2 "annual refresh =
  YAML edit not code change" doctrine.
- `VerdictResult` is a Pydantic model with `level`, `headline_reason`,
  `details: list[str]`.

Confidence: MEDIUM. Thresholds are debatable (43% DTI is the heuristic
ATR/QM line per Phase 2 ATR-QM note: price-based General QM since 2022
dropped the hard 43% cap, but it's still the conventional UW line).
Lock in /gsd-discuss-phase via Open Question #Q2.

### Pattern 5: DuckDB `analyzed_listings` table

**Module:** `orchestration/init-db.mjs` gets two new DDL statements +
`cmdInsertAnalyzedListing` added to `orchestration/db-write.mjs`.

```sql
-- Add after the existing `properties` block.
CREATE SEQUENCE IF NOT EXISTS analyzed_listings_id_seq START 1;

CREATE TABLE IF NOT EXISTS analyzed_listings (
  id            INTEGER PRIMARY KEY DEFAULT nextval('analyzed_listings_id_seq'),
  zpid          BIGINT  NOT NULL,
  source_url    VARCHAR NOT NULL,
  list_price    DECIMAL(14,2) NOT NULL,
  zip           VARCHAR(5) NOT NULL,
  property_type VARCHAR NOT NULL,
  beds INTEGER, baths DECIMAL(3,1), sqft INTEGER, year_built INTEGER,
  zestimate DECIMAL(14,2),
  property_tax_annual DECIMAL(14,2),
  hoa_monthly DECIMAL(14,2),
  listing_json  JSON NOT NULL,       -- full PropertyListing record
  analysis_json JSON NOT NULL,       -- full PropertyAnalysisReport
  verdict       VARCHAR NOT NULL CHECK (verdict IN ('GO','WATCH','NO_GO')),
  fetched_at    TIMESTAMP NOT NULL,
  analyzed_at   TIMESTAMP NOT NULL DEFAULT now()
);

CREATE INDEX IF NOT EXISTS idx_analyzed_listings_zpid
  ON analyzed_listings(zpid);
CREATE INDEX IF NOT EXISTS idx_analyzed_listings_verdict
  ON analyzed_listings(verdict);
CREATE INDEX IF NOT EXISTS idx_analyzed_listings_analyzed_at
  ON analyzed_listings(analyzed_at);
```

**Decisions:**

- `zpid` is NOT the PK — a listing can be analyzed multiple times
  (price drops, weeks apart). Surrogate `id` is the PK; `zpid` is
  indexed for watchlist queries. Matches existing `scenarios` table.
- `listing_json` + `analysis_json` are DuckDB native JSON columns
  matching Phase 9 D-03-05 (`scenarios.request_json` + `response_json`).
- Decimal columns are read via `CAST AS VARCHAR` and written as JSON
  strings per Phase 9 D-03-02 / D-03-03.
- Write path: `node orchestration/db-write.mjs insert-analyzed-listing
  --json /tmp/listing-and-report.json` wrapped in `withLock()` exactly
  like every other write subcommand (PERS-05).

**Watchlist query examples (v1.2 watchlist mode consumes these):**

```sql
-- Last 30 days, GO verdict, under $900k
SELECT zpid, source_url, list_price, verdict, analyzed_at
FROM analyzed_listings
WHERE analyzed_at >= now() - INTERVAL 30 DAY
  AND verdict = 'GO' AND list_price < 900000
ORDER BY analyzed_at DESC;

-- Repeated re-analyses of the same property (price tracking)
SELECT zpid, list_price, verdict, analyzed_at
FROM analyzed_listings WHERE zpid = $1
ORDER BY analyzed_at ASC;
```

Confidence: HIGH (mirrors Phase 9 patterns exactly).

### Pattern 6: SKILL.md `property` mode routing

**Mode dispatch table addition** (in SKILL.md `## Mode Routing`):

```
| Zillow URL OR "analyze this listing" OR "what would buying this look
  like" | `property` | `scripts/property_analyze.py` |
```

**Precedence ordering** (insert before existing rules):

```
0. Zillow URL substring (`zillow.com/homedetails/` or `zillow.com/b/`)
   → `property` (HIGHEST — URL pin overrides all verb routing)
1. Explicit sub-command           → `/mortgage-ops {mode}`
...  (existing order continues)
```

"URL pin overrides verbs" prevents mis-routing when the user types
"should I refi into this listing https://zillow.com/..." — the URL is
the load-bearing signal.

**Mode file scaffold (`modes/property.md`)** — mirror `evaluate.md`
structure. Sections:

1. `## When to invoke` — 3-4 example prompts
2. `## What scripts to call` — single script `scripts/property_analyze.py`
3. `## Listing ingestion subroutine`
   - WebFetch call with the extractor prompt (Pattern 1)
   - Pydantic validate-only round-trip
   - Interactive gap-fill loop (one question per missing critical field)
   - Final input JSON to `scripts/property_analyze.py`
4. `## What to narrate`
   - `_shared.md` "answer + provenance" template
   - Multi-program table (Pattern 7) with verdict at the top
5. `## Edge cases` — 403/captcha, missing `__NEXT_DATA__`, missing
   list_price, missing zip
6. `## Persist` — `node orchestration/db-write.mjs
   insert-analyzed-listing`; emit `reports/{NNN}-property-{zpid}-{YYYY-
   MM-DD}.md` (Pattern 7)
7. `## RELATED REFERENCES` (load on demand)

**Token budget:** adding `property` mode adds ~12-15 lines to SKILL.md
(routing row + precedence row + one-paragraph callout). SKILL.md is
currently 278 lines / ~3,400 cl100k tokens; v1.1 fits within the
500-line / 5K-token ceiling.

Confidence: HIGH (mode-file shape and dispatch additions are
mechanical).

### Pattern 7: Report formatter

**Module:** `lib/property_report.py` (new). Pure renderer; takes a
`PropertyAnalysisReport` and emits markdown.

**Why not in `scripts/property_analyze.py` directly:** the script
returns JSON; the skill narrates. But for the "save to file" flow
matching Phase 11 amortization-agent's CSV-path contract, we also need
a markdown emitter that can be persisted directly to
`reports/{NNN}-property-{zpid}-{YYYY-MM-DD}.md` without round-tripping
through Claude. So `lib/property_report.py` provides:

```python
def render_report_markdown(report: PropertyAnalysisReport) -> str: ...
def render_report_csv(report: PropertyAnalysisReport) -> str: ...  # v1.2
```

The script gets a `--format {json,markdown}` flag. Default is `json`
(skill narrates); `--format markdown --output reports/...md` writes
the file and prints the path to stdout (mirrors Phase 11
amortization-agent's CSV-path contract).

**Markdown report sections (sketch — concrete template ships
in implementation phase):**

```markdown
# Property Analysis: {address}
*Generated {analyzed_at} from {source_url}*

## Verdict: **GO** / **WATCH** / **NO_GO**
> {headline_reason}
- {detail 1}
- {detail 2}

## Listing snapshot
| Field | Value | Source |
| List price / Zestimate | $789,000 / $762,400 *(estimated)* | scraped/estimated |
| Tax / HOA / Beds-Baths-Sqft / Year / Type | ... | mixed provenance |

## Program eligibility
(table: Program, Limit, Price vs limit, Eligible — one row per program)

## Down-payment sweep (Conv30, 6.875% MORTGAGE30US)
(table: Down %, Down $, Loan $, P&I, PMI, PITI, DTI back, LTV,
Cash-to-close — one row per down-payment scenario in
DOWN_PAYMENT_SCENARIOS; FHA-only / 15yr / VA / Jumbo rendered as
additional matrices below)

(table: Scenario, New rate, New P&I, New DTI back, Verdict — baseline +
+100bps + +200bps + -25% income)

## Refi opportunity (if rates fall to MORTGAGE30US - 100bps)
- New P&I, monthly savings, closing costs estimate, simple breakeven,
  5-year NPV (one bullet each)

## Tax deductibility (IRS Pub 936)
- Under $750k cap (post-2017): yes/no
- Estimated annual interest year-1
- Estimated marginal tax savings at profile.marginal_tax_rate

## Provenance
*Every dollar figure traces to a script invocation. Listing data came
from {extraction_method}. User-provided fields tagged inline.*
```

The Phase 11 amortization-agent contract (CSV path under
`reports/{NNN}-...csv` OR markdown table to stdout) is the precedent —
mirror it. **v1.1 emits BOTH** markdown for chat narration AND saves
markdown to `reports/` when `save_report: true` in `_profile.md`. CSV
emission is OUT of v1.1 scope; the matrix is shaped for markdown
tables.

Confidence: HIGH (contract mirrors Phase 11 SUBA-04 exactly).

### Pattern 8: Test fixtures (pinned Zillow snapshots)

**Layout:**

```
tests/fixtures/zillow/
  {slug}.html                       # pinned raw HTML, ~100KB after strip
  {slug}.next_data.json             # extracted __NEXT_DATA__ blob
  {slug}.expected_listing.json      # golden PropertyListing
  {slug}.expected_report.md         # golden formatter output
where {slug} ∈ {sfh-conventional-eligible, condo-fha-eligible,
                 jumbo-bay-area, multifamily-2unit, captcha-block}
  captcha-block: only .html — extraction MUST fail loud
```

**Why pinned HTML over live fetches:**

1. Phase 12 EVAL doctrine: deterministic test inputs only. Live
   WebFetch against Zillow in CI is forbidden by the same logic that
   bans live FRED in CI (D-12-LIVE01-* "synthetic-only-in-CI" policy).
2. Zillow's HTML rotates frequently; pinning anchors the extractor
   against a known input.
3. Pinned HTML lets us regression-test the captcha-detection path
   (`captcha-block.html`) without ever needing to actually trigger
   PerimeterX.

**Fixture selection criteria (5 covers all branches):**

| Fixture | Programs eligible | Price tier | Type | Special |
|---|---|---|---|---|
| sfh-conventional-eligible | Conv 30/15, FHA | $400k | SFH | Default |

3. Sanitize PII: redact agent phone numbers, full street addresses if
   the household doesn't want them in the repo. Synthetic addresses
   preserve zip.
4. Commit. The fixtures become regression baseline.

the freezegun fixture-pinning idiom; v1.1 inherits both).

## Pitfalls (the looks-done-but-isn't list)

### Pitfall 1: WebFetch returns HTTP 200 but body is a captcha page

PerimeterX often serves a captcha challenge with a 200 status. HTML
contains `px-captcha`, `Press & Hold to confirm you are a human`, or
`perimeterx.net`. A naive extractor sees "200, parse JSON" and emits a
half-extracted result with mostly-null fields; the analysis silently
degrades.

**Mitigation:** The WebFetch extractor prompt (Pattern 1) MUST look for
those substrings FIRST and return `{"_block_detected": true,
"signal_phrase": "<which matched>"}` before attempting `__NEXT_DATA__`
extraction. Mode handler in `modes/property.md` branches on
`_block_detected` and switches to manual mode.

### Pitfall 2: Zestimate 5-15% off list price; using it as ground truth is wrong

Zillow's algorithmic Zestimate can diverge from the seller's list price
by 5-15% either direction. Both should be reported; the **list price is
the underwriting input** (banks lend against list/contract price, not
Zestimate). A property might compute as "WATCH" against Zestimate but
"GO" against list (or vice versa).

**Mitigation:** `PropertyListing.list_price` is required;
`zestimate` is optional and never flows into eligibility math. Report
shows Zestimate purely for narrative context — "list $789k vs
Zestimate $762k (~3.5% above Zillow's model)" — with `*(estimated)*`
labels per Phase 7 SKLL-APR-1 precedent (Pitfall 8 below).

### Pitfall 3: HOA $0 vs null ambiguity

Some condos legitimately have $0 HOAs (rare); others have unreported
HOAs because the listing agent didn't enter it. Treating null as $0
silently breaks PITI for ~20% of condo analyses.

**Mitigation:** `PropertyListing.hoa_monthly` is `ProvenancedMoney`,
not `Money | None`. When `value=None`, `provenance="missing"`. Gap-fill
ALWAYS asks the user: "I couldn't find HOA data. Type the monthly $
amount, or 'none' if there's no HOA." User types 'none' → store
`value=Decimal("0.00"), provenance="user_provided"`. `$0` with
`provenance=zillow_scraped` is FORBIDDEN — we never silently default;
we always ask.

### Pitfall 4: Property tax annual vs monthly ambiguity

Zillow's `__NEXT_DATA__` field for property tax is typically the
annual amount as reported by county records, but some agent-entered
fields show monthly. PITI computation divides annual / 12; if the
field is already monthly, we'd be dividing twice and undershooting
PITI by ~$700/mo on a $800k house.

**Mitigation:** Extract BOTH `propertyTaxRate` (% of assessed) and
`taxAnnualAmount` from `__NEXT_DATA__`. If both exist, cross-check via
`abs(taxAnnualAmount - propertyTaxRate * list_price) / taxAnnualAmount
< 0.50`. If they diverge by >50%, flag the listing and trigger
gap-fill. If only one exists, use it as-is and surface its provenance
in the report.

### Pitfall 5: Conforming loan limit varies by county AND by unit count

A 2-unit duplex at $850k in a baseline county is jumbo (over $832,750
1-unit baseline) but NOT jumbo when treated as 2-unit ($1,066,250
2-unit baseline). The `lib.rules.loan_type.classify()` predicate
already takes `unit_count` (per Phase 2 conforming-limits-2026.yml
`MissingCountyDataError` for unit_count>1); v1.1 MUST pipe
`property_type=Multifamily` through to `unit_count=2..4` correctly.

**Mitigation:** Add a defensive test fixture (`multifamily-2unit.html`)
that exercises the unit-count branch. Current YAML only ships
`one_unit` data for high-cost counties per Phase 2 BL-04 — accept the
existing `MissingCountyDataError` and surface it to the user ("This is
a 2-unit property in {county}; I don't have 2-unit limit data for that
county. Type the limit manually if you know it.").

### Pitfall 6: Jumbo cutoff is sliding (baseline vs ceiling)

2026 conforming is $832,750 baseline / $1,249,125 ceiling. "Jumbo"
means above the applicable limit IN THAT COUNTY. A $900k loan is
conforming in any high-cost county but jumbo in baseline counties.
Hardcoding "$832,750 = jumbo" is wrong for ~232 high-cost counties
nationwide (the Pachulski household is in WA, where King County is
on the high-cost list).

**Mitigation:** Already handled by `lib.rules.loan_type.classify`
(Phase 2). v1.1 just consumes the verdict. The report formatter MUST
show the EFFECTIVE limit for the listing's county, not the national
baseline. Render row: "Conforming limit for {county_name}: $1,249,125
(high-cost area)."

### Pitfall 7: VA eligibility comes from the household, not the property

`profile.yml` carries `va_eligible: bool` (per Phase 4 affordability
convention). Listings don't carry VA-eligibility; the household does.
The mode handler MUST read profile before deciding whether to run a
`va_30` cell — never default to "eligible" or "ineligible" without
checking.

**Mitigation:** `lib/property_analysis.analyze_property` takes the
parsed `profile.yml` as a function arg and reads `profile.va_eligible`
explicitly. If the field is missing from the user's profile.yml, emit
a 6-key Pydantic envelope error pointing to the example profile.

### Pitfall 8: "Estimated" labels on Zillow-sourced numbers

Every Zillow-sourced number that flows into the report (Zestimate, tax
estimate, insurance estimate) MUST carry the `estimated` literal text
in the narration layer per Phase 7 SKLL-APR-1 precedent extended to
v1.1 scope. Forbidden: "tax $9,468/yr" → must be "$9,468/yr
(estimated)".

**Mitigation:** Bind a new rule SKLL-EST-1 in SKILL.md: "Every
Zillow-sourced money figure displays with `*(estimated)*` suffix when
`provenance ∈ {zillow_scraped, zillow_estimated}` and the field is
known agent-estimated (tax, insurance, Zestimate). List price is the
exception — it's the literal asking price, not an estimate."

### Pitfall 9: WebFetch truncates to 100KB for the Haiku sub-prompt

WebFetch docs say max fetch ~10MB but truncates to ~100KB for the
secondary Haiku conversation. A Zillow listing with a long description,
agent bio, and embedded photo metadata can be 2-3MB raw HTML. After
conversion to markdown and truncation, the `__NEXT_DATA__` block
(typically ~30-80KB) may fall off the end.

**Mitigation:** Extractor prompt (Pattern 1) instructs Haiku to "look
for `__NEXT_DATA__` FIRST before reading any other content; if
truncated before it appears, return `{"_truncated": true}`." Mode
handler treats `_truncated:true` like a captcha block — switch to
manual mode.

### Pitfall 10: Mode routing collision with `evaluate` or `refinance`

A prompt like "Should we refi to buy this listing? https://zillow.com/
..." combines a refi verb (`refinance` mode) and a Zillow URL
(`property` mode). The Pattern 6 precedence rule says URL-pin wins, but
the user's actual intent might be "use refi math against this listing's
price" — a different workflow.

**Mitigation:** When `property` mode is selected via URL-pin but a refi
or affordability verb appears in the same prompt, ASK ONE clarifying
question (per existing SKILL.md disambiguation contract): "I see a
Zillow URL AND 'refi' — did you mean (a) analyze this property fresh
[property mode], or (b) treat the Zillow price as a refi target
[refinance mode]?"

### Pitfall 11: ZPID collisions across reanalyses

A user analyzing the same listing every Saturday during a 3-month
search will accumulate 12+ rows in `analyzed_listings` for the same
zpid. The DuckDB design intentionally allows this (Pattern 5 surrogate
`id` PK), but the report formatter MUST NOT silently include stale
prior analyses; each analysis is self-contained.

**Mitigation:** When persisting, check for prior analyses of the same
zpid and surface a "you've analyzed this property N times since
{first_seen}; latest list_price was $X" callout. Makes price tracking
emergent without building a separate watchlist feature in v1.1.

### Pitfall 12: Single-writer DuckDB contention on parallel analyses

If a user kicks off two analyses in parallel (two Zillow URLs in the
same chat session), both Node `db-write.mjs` invocations hit the
lockfile. Phase 9 PERS-04 already handles 60s stale recovery, but two
~1-second writes back-to-back will queue, not collide.

**Mitigation:** No new work; Phase 9 PERS-04/05 already covers this.
Surface in the property mode body: "Analyses are serialized; if you
queue multiple URLs, expect modest sequencing between writes."

## Open Questions (for /gsd-discuss-phase to lock)

### Q1: Personal vs generic baseline scenarios

Should the report show YOUR scenarios (using `profile.yml` +
`household.yml` — actual income, debts, credit, reserves,
VA-eligibility) or a generic 20%-down baseline ("here's what a typical
buyer would see")? The personal version is what makes mortgage-ops
differentiated. The generic version is useful when sharing the report
with a partner who hasn't loaded their profile yet.

**Recommendation:** personal by default; add a `--generic` flag.

### Q2: Verdict thresholds — where do 43% DTI / 30% rate-shock / 45% post-shock come from?

Pattern 4 picks numbers that match conventional UW heuristics but
aren't bright-line regulatory. The Phase 2 ATR/QM note says the 43%
cap is "GONE — surface as heuristic only." Should defaults match
conventional UW heuristics (43/30/45) or the household's personal risk
tolerance (which might be tighter: 35/20/40)?

**Recommendation:** ship conventional UW defaults in
`property-analysis-heuristics.yml`; profile.yml can override per knob.

### Q3: Default stress scenarios — which ones run automatically?

Pattern 3 currently fans out 2 rate shocks (+100bps, +200bps) and 1
income shock (-25%) = 3 rows. Could be more (+50/+100/+200 plus -10
and -25 = 5 rows) or fewer.

**Recommendation:** 2 rate + 1 income = 3 rows default; profile.yml
override.

### Q4: Refi opportunity rate — FRED MORTGAGE30US - 100bps, or × 0.85, or some other indicative drop?

Pattern 3 sketches `market_rate_30yr - 0.01` (100bps drop). Could also
be `× 0.85` (slightly larger relative drop) or "compare against
historical MORTGAGE30US 5-year low." Hardcoded -100bps is simplest and
most honest about its heuristic nature.

**Recommendation:** hardcode -100bps; surface a
`--refi-rate-delta 50bps` flag for tuning.

### Q5: Down-payment scenarios — which 6?

Pattern 3 ships 3.5%/5%/10%/15%/20%/25%. Could be 3%/5%/10%/15%/20%/25%
(conventional 97 minimum 3% vs FHA 3.5%). Or fewer rows
(3.5%/10%/20%).

**Recommendation:** 3.5%/5%/10%/15%/20%/25% — matches FHA minimum and
is standard rate-sheet stagger.

### Q6: Verdict copy ergonomics

GO / WATCH / NO_GO are clinical. User may want softer copy ("Strong
fit" / "Caution" / "Not a fit"). What ships as default?

**Recommendation:** GO/WATCH/NO_GO internal labels; user-facing
narration uses "Strong fit" / "Watch list" / "Not viable" with
profile.yml `verbosity` controlling whether the headline reason is
one line or three.

### Q7: Apify (or other paid scraper) integration — v1.1 or v1.2?

If WebFetch's hit rate against Zillow degrades during v1.1 development
(PerimeterX evolves; `__NEXT_DATA__` access pattern rotates), do we
ship a paid-scraper fallback in v1.1 (delays milestone) or v1.2?

**Recommendation:** v1.2. WebFetch + gap-fill is v1.1 contract. v1.2
adds Apify (`maxcopell/zillow-detail-scraper`, $1.70/1k results) behind
a `--scraper apify` flag if WebFetch degradation becomes chronic.

### Q8: Watchlist mode — v1.2 stretch or v1.1?

Pattern 5's `analyzed_listings` table is shaped for a watchlist UX
("show me every GO under $900k from the last 30 days"). Building the
mode is ~1 phase of work.

**Recommendation:** v1.2. v1.1 is "one URL → one report." Watchlist
is a different UX (no URL input; pure DuckDB query) and deserves its
own mode.

   `_synthesize_verdict()` + `data/reference/property-analysis-
   heuristics.yml` + `data/reference/insurance-estimate-defaults.yml`.
4. **Phase 16: Report formatter.** Ships `lib/property_report.py` with
   markdown emission + 5 golden-value `expected_report.md` fixtures
   matching the 5 pinned HTML inputs.
5. **Phase 17: DuckDB persistence.** Ships the `analyzed_listings`
   schema migration + `cmdInsertAnalyzedListing` in db-write.mjs +
   integration tests against pre-pinned fixture inputs.
